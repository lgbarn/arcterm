# INTEGRATIONS.md

## Overview

ArcTerm integrates with SSH (via two parallel backends: libssh-rs and libssh2), tmux (via a control-mode protocol parser), the system font stack (FreeType/HarfBuzz/platform locators), a dual GPU rendering backend (wgpu/WebGPU and Glium/OpenGL), a Lua 5.4 scripting and plugin system, OS notification services (macOS UNNotifications, Windows toast, Linux D-Bus), and the system clipboard. No AI/LLM integrations exist yet; they are planned. The mux protocol operates over Unix domain sockets and optionally TLS TCP, using a custom binary framing format.

## Findings

### SSH Integration

- **Two SSH backends**, both optional and selected by feature flags, with `default = ["libssh-rs", "ssh2"]` (both enabled):
  - **libssh-rs** (v0.3.6): Rust bindings to libssh, compiled with `features=["vendored"]` so the C library is bundled.
  - **ssh2** (v0.9.3): Rust bindings to libssh2 (classic). Enabled with `features=["openssl-on-win32"]`.
  - Evidence: `wezterm-ssh/Cargo.toml` lines 13-16, 30-38
- The `wezterm-ssh` crate (v0.4.0) is the unified SSH abstraction layer that wraps both backends behind a common `Session` API.
  - Evidence: `wezterm-ssh/src/lib.rs` (exports `Session`, `SessionEvent`, `SftpRequest`, etc.)
  - Evidence: `wezterm-ssh/src/session.rs` (`SessionEvent` enum: Banner, HostVerify, Authenticate, Authenticated, Error)
- **SFTP** is supported. A dedicated `sftp/` module exists alongside channel, pty, auth, and host modules.
  - Evidence: `wezterm-ssh/src/sftp/` directory listing
- **SSH domain type** in the mux: `RemoteSshDomain` wraps `SshDomain` config, allowing SSH sessions to appear as mux domains with panes and tabs.
  - Evidence: `mux/src/ssh.rs` lines 1-28; `wezterm-gui/src/main.rs` lines 136-178
- **SSH multiplexing**: Configurable via `SshMultiplexing` enum. In the `run_ssh` path, it is set to `SshMultiplexing::None`.
  - Evidence: `wezterm-gui/src/main.rs` line 151
- **Password/key prompt**: Rendered inside a terminal widget using termwiz's `LineEditor` with masked echo.
  - Evidence: `mux/src/ssh.rs` lines 31-58
- **Key material**: OpenSSL handles TLS; `async_ossl` (local path crate) provides async wrappers. `sha2` and `hex` handle fingerprint operations.
  - Evidence: `wezterm-ssh/Cargo.toml` lines 38, 44

### Multiplexer Protocol

- **Protocol**: A custom binary PDU framing protocol (not protocol buffers, not msgpack). Frames carry: length (LEB128), serial number (LEB128), PDU identifier (LEB128), then payload serialized with `varbincode`.
  - Evidence: `codec/src/lib.rs` lines 1-56
- **Compression**: zstd (v0.11) is used for large payloads; a `COMPRESSED_MASK` bit flag on the length field indicates compression.
  - Evidence: `codec/src/lib.rs` line 58; `Cargo.toml` line 276
- **Codec version**: `CODEC_VERSION = 45` — bumped on backwards-incompatible changes.
  - Evidence: `codec/src/lib.rs` line 444
- **PDU types** (selected, from the `pdu!` macro at lines 450-505 of `codec/src/lib.rs`):
  - `Ping/Pong`, `ListPanes/ListPanesResponse`, `WriteToPane`, `SendKeyDown`, `SendMouseEvent`, `GetLines`, `SetClipboard`, `GetTlsCreds/GetTlsCredsResponse`, `SpawnV2`, `SplitPane`, `KillPane`, `SearchScrollbackRequest`, `GetImageCell`, `MovePaneToNewTab`, `AdjustPaneSize`
  - Evidence: `codec/src/lib.rs` lines 450-505
- **Transport options**:
  - **Unix domain socket**: Primary local transport. Socket path stored as `WEZTERM_UNIX_SOCKET` env var.
    - Evidence: `wezterm-gui/src/main.rs` line 414 (`gui-sock-{pid}`)
  - **TLS over TCP**: Available for remote connections. Per-session CA + client certs generated by `rcgen`.
    - Evidence: `wezterm-mux-server-impl/src/pki.rs` (full PKI bootstrap); `codec/src/lib.rs` line 629 comment
- **Discovery**: `wezterm-client/src/discovery.rs` handles publishing and resolving the GUI socket path.
  - Evidence: `wezterm-gui/src/main.rs` lines 519-526

### tmux Integration

- **tmux control mode (CC) protocol**: Parsed via a PEG grammar using `pest` (v2.7). The feature is behind the `tmux_cc` feature flag of the `termwiz` crate (enabled by default).
  - Evidence: `termwiz/Cargo.toml` lines 29-30, 56 (`tmux_cc = ["pest", "pest_derive", ...]`)
- **tmux domain**: `TmuxDomain` / `TmuxDomainState` in `mux/src/` represent a tmux session as a mux domain, mapping tmux sessions/windows/panes to ArcTerm tabs/panes.
  - Evidence: `mux/src/tmux_commands.rs` lines 1-51 (`PaneItem`, `WindowItem` structs, `TmuxDomainState`)
- **Command dispatch**: The `TmuxCommand` trait abstracts individual tmux CC commands (list-all-panes, resize, etc.) with `get_command()` / `process_result()` methods.
  - Evidence: `mux/src/tmux_commands.rs` lines 20-23

### Font Rendering Pipeline

The font pipeline has four distinct abstracted layers, each with platform-specific implementations behind trait objects:

1. **Font Locator** (discovers fonts on the system):
   - `font_config.rs` — Linux/Android (fontconfig)
   - `core_text.rs` — macOS (Core Text)
   - `gdi.rs` — Windows (DirectWrite via `dwrote`)
   - Evidence: `wezterm-font/src/locator/` directory
2. **Font Parser** (`wezterm-font/src/parser.rs`): Reads font files, extracts metadata, maps to `ParsedFont`.
3. **Font Shaper** (text shaping — glyph selection, ligatures, BiDi):
   - HarfBuzz shaper (primary): `wezterm-font/src/shaper/harfbuzz.rs`
   - Evidence: `wezterm-font/src/shaper/` directory
4. **Font Rasterizer** (glyph pixel rendering):
   - FreeType rasterizer: `wezterm-font/src/rasterizer/freetype.rs`
   - HarfBuzz rasterizer (COLR/color fonts): `wezterm-font/src/rasterizer/harfbuzz.rs`
   - COLR support: `wezterm-font/src/rasterizer/colr.rs`
   - Evidence: `wezterm-font/src/rasterizer/` directory

- **Font database**: SQLite (via rusqlite) used for caching font metadata.
  - Evidence: `wezterm-font/src/db.rs` (filename implies font DB)
- **Emoji support**: `emojis` (v0.6) and `finl_unicode` (git fork) handle emoji categorization and grapheme cluster splitting.
  - Evidence: `Cargo.toml` lines 84, 95; `wezterm-gui/Cargo.toml` lines 50, 53

### GPU / Rendering Backend

- **Dual backend architecture**: The `RenderContext` enum routes rendering calls to either the Glium (OpenGL) or WebGPU backend.
  - Evidence: `wezterm-gui/src/renderstate.rs` lines 23-31
- **wgpu (v25.0.2)**: WebGPU implementation; state managed in `WebGpuState`. Uses a WGSL shader at `wezterm-gui/src/shader.wgsl`.
  - Evidence: `wezterm-gui/src/termwindow/webgpu.rs` lines 24-38
- **Glium (v0.35)**: Legacy OpenGL backend; GLSL shaders at `wezterm-gui/src/glyph-frag.glsl` and `glyph-vertex.glsl`.
  - Evidence: `wezterm-gui/src/renderstate.rs` lines 7-11
- **Texture atlas**: `guillotiere` (v0.6) manages glyph texture atlas allocation.
  - Evidence: `Cargo.toml` line 113
- **Image protocol**: Supports inline images (Kitty protocol and iTerm2 protocol). `wezterm-blob-leases` manages image blob storage via a temp-dir backend.
  - Evidence: `wezterm-gui/src/main.rs` lines 416-418; `termwiz/Cargo.toml` feature `use_image`

  - Evidence: `wezterm-gui/src/main.rs` line 74

  - Evidence: `config/src/lua.rs` lines 23-31 (`SETUP_FUNCS` registry); `wezterm-gui/src/main.rs` lines 1205-1207

  - Evidence: `config/src/lua.rs` lines 75-100
- **Live reload**: `notify` (v5.0.0) watches config file changes and triggers reload.

### Plugin System

- **Mechanism**: Git-based. The `plugin` Lua API crate clones or updates git repositories into a plugins directory using `git2` (v0.20).
  - Evidence: `lua-api-crates/plugin/src/lib.rs` lines 1-9 (imports `git2::Repository`, `Remote`)
- **Install path**: A per-user plugins directory computed from `RepoSpec::plugins_dir()`.
  - Evidence: `lua-api-crates/plugin/src/lib.rs` line 62
- **Directory naming**: Git URLs are encoded into filesystem-safe directory names using a deterministic encoding scheme (slashes become `sZs`, colons `sCs`, dots `sDs`).
  - Evidence: `lua-api-crates/plugin/src/lib.rs` lines 20-42
- **No WASM sandbox**: The planned `arcterm-wasm-plugin` crate (capability-based WASM sandbox) does not exist yet. Plugins are Lua modules loaded directly into the interpreter.
  - Evidence: `Cargo.toml` members list (no `arcterm-wasm-plugin`); `CLAUDE.md` "Planned ArcTerm Extensions"

### WSL Integration (Windows)

- **Auto-discovery**: On Windows, `config/src/wsl.rs` calls `wsl.exe -l -v` and parses the UTF-16 output to enumerate installed WSL distributions.
  - Evidence: `config/src/wsl.rs` lines 19-78
- **WslDomain**: Each distribution becomes a mux domain.
  - Evidence: `config/src/wsl.rs` line 8

### TLS / PKI for Remote Mux

- A fresh CA and server certificate are generated each time the mux server starts (using `rcgen` v0.12).
- The client obtains its cert through the Unix domain socket channel, then connects to TLS TCP.
- DNS lookup (`dns-lookup` v2.0) is used to populate TLS certificate SANs.
  - Evidence: `wezterm-mux-server-impl/src/pki.rs` lines 1-50; `Cargo.toml` line 81

## Open Questions

- The update check endpoint (`https://api.github.com/repos/wezterm/wezterm/releases/latest`) still points to the upstream wezterm repository. Should this be updated to an ArcTerm release endpoint, or disabled until ArcTerm has its own release infrastructure?
- The plugin system uses `git2` without any sandboxing — Lua plugins execute with full process privileges. Is a capability model planned before making this user-facing?
- The dual GPU backend (Glium + wgpu) means two code paths for every render operation. Is there a plan to deprecate Glium once wgpu is mature and fully tested on all platforms?
- `reqwest` v0.12 appears in the workspace dependencies but no first-party crate was observed importing it directly. Which transitive dependency pulls it in?
